# C++游戏服务端转Java重写设计方案

**项目名称**：Priston Tale 服务端重写
**日期**：2026-03-17
**目标**：将C++游戏服务端100%兼容重写为Java版本

## 一、项目概述

### 1.1 背景信息
- **C++原版**：位于 `PristonTale-EU-main/Server/` 目录
- **Java目标**：位于 `jpstale-server` 目录
- **当前状态**：pt-common已实现原版消息加解密，可正常处理登录请求和响应

### 1.2 重写目标
- 100%重写所有核心功能（登录、游戏逻辑、公会系统等）
- 完全兼容原版协议（客户端可无缝连接）
- 采用增量式开发，先基础框架再逐步添加功能
- 仅支持PostgreSQL数据库

### 1.3 技术架构
```
Java服务端体系
├── pt-common          # 协议层、网络层、工具类
├── pt-dao            # 数据访问层（PostgreSQL多schema）
├── pt-login-server   # 登录服务器（端口10009）
├── pt-game-server    # 游戏服务器（端口10007）
└── pt-web-server     # Web服务器（端口8080）
```

## 三、关键技术决策

### 3.1 协议兼容性
- 完全遵循C++包格式（iLength + iHeader + body）
- 保持相同的加密算法和密钥
- 一致的错误码和响应格式

### 3.2 数据存储
- PostgreSQL单库多schema（userdb、gamedb、clandb等）
- 使用MyBatis-Plus简化数据访问
- 保持与原版数据库结构一致

### 3.3 性能考虑
- 使用Netty作为网络框架
- 合理的线程池配置
- 缓存热点数据

### 3.4 可观测性
- 完整的日志记录
- 性能监控指标
- 便于调试的接口

## 四、验证策略

每个阶段完成后进行：
1. **单元测试**：覆盖核心逻辑
2. **集成测试**：验证模块间交互
3. **兼容性测试**：使用原版客户端验证

## 五、风险控制

1. **协议差异**：通过源码分析确保协议一致性
2. **无法抓包**：通过源码分析和测试工具弥补
3. **性能问题**：提前进行压力测试
4. **数据一致性**：编写数据校验工具

## 六、启动顺序

1. 先启动 pt-login-server（10009）
2. 再启动 pt-game-server（10007）
3. 最后启动 pt-web-server（8080，包含Redis依赖）

## 七、依赖关系

- **pt-common**：被所有需要协议/编解码的模块依赖
- **pt-dao**：通过 pt-dao-spring-boot-starter 提供数据访问
- **pt-web-server**：依赖 Redis 存储登录会话
